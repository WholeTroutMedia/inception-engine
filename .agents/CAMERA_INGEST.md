## Purpose

CAMERA_INGEST is the dedicated agent responsible for receiving camera-originated files the instant they arrive on the network and bringing them under the Creative Liberation Engine's governance. It is the first constitutional checkpoint that media passes through on its way from shutter to mesh.

This agent exists because raw camera output — whether from a Sony A1 II, a cinema camera, or any future device — arrives as ungoverned data. It has no name the system recognizes, no contract applied, no provenance recorded, and no downstream agents aware of its existence. CAMERA_INGEST transforms ungoverned bytes into governed assets.

## Primary Duties

1. **Receive** new camera-originated files from edge sources (Sony A1 II via FTP/FTPS/SFTP, monitored directories on NAS or workstation shares, SD card mount points).

2. **Normalize** filenames and directory placement according to the active Storage Contract for the target pool. Raw camera filenames (e.g. `DSC00001.ARW`) become canonical identifiers that encode device, date, sequence, and context.

3. **Extract and enrich metadata** — EXIF, maker notes, device ID, timestamps, GPS where available, lens and exposure data. This metadata becomes part of the asset's permanent record.

4. **Compute checksums** (SHA-256) for integrity verification. The checksum is recorded at ingest time and is the canonical proof that the file has not been altered since capture.

5. **Write an ingest record** to the canonical manifest or log store. This record is the asset's birth certificate within the system.

6. **Emit a structured ingest event** onto the Mesh for downstream consumers — thumbnailers, AI taggers, sync/backup agents, delivery pipelines, and any future agent that needs to know a new asset has arrived.

## Inputs

### Transport Protocols
- FTP/FTPS/SFTP endpoint exposed on a mesh-connected node
- Monitored directories on NAS or workstation shares (filesystem watch)
- SD card mount-point detection (future)

### Supported File Types
- RAW: ARW (Sony), CR3 (Canon), NEF (Nikon), DNG
- Compressed: JPEG, HEIF/HEIC
- Video: MP4, MOV, MXF
- Sidecar: XMP
- Proxy: JPEG/MP4 proxy files generated by camera

### Configuration Source
- Storage Contract reference (e.g. `storage-registry/camera-a1ii.yml`)
- Pool identifier (e.g. `NAS_PRIMARY`, `WS_JUSTIN`, `CLOUD_BUCKET_X`)

## Outputs

### 1. Normalized File Paths
Files are moved/renamed into canonical locations within the storage pool:

```
/photos/{year}/{month}/{day}/{device}/{event_slug}/{DEVICE}_{YYYYMMDD}_{HHMMSS}_{SEQ}.{EXT}
```

Example:
```
/photos/2026/03/10/a1ii/portrait-session/A1II_20260310_145732_0001.ARW
```

## Behaviors

### On File Arrival
1. Detect new file in watched directory or via FTP upload completion.
2. Wait for file write to complete (size stability check over 2-second window).
3. Validate against Storage Contract (allowed types, size limits, expected directory).
4. Compute SHA-256 checksum.
5. Generate logical_id from device + date + sequence.
6. Move/rename into final canonical location per contract naming rules.
7. Extract EXIF and metadata.
8. Persist ingest record to manifest store.
9. Emit `camera.ingest.v1` event onto Mesh.

### On Contract Update
- Watch for changes to the relevant Storage Contract in Git.
- Reload rules without manual restart where possible.
- Optionally run migrations on newly ingested assets if rules change (e.g. new naming pattern).
- Log contract version transitions for audit.

### On Error
- Quarantine problematic files into a clearly marked directory: `/quarantine/camera-ingest/{date}/{filename}`
- Record error reason alongside quarantined file.
- Emit a `camera.ingest.error.v1` event for governance agents.
- Never silently drop a file. Every file that enters the watch path must either be ingested or quarantined. No third state.

### On Device Discovery
- When a new camera device ID appears for the first time, emit a `camera.device.discovered.v1` event.
- Log the device with its maker notes and serial number.
- If no Storage Contract exists for this device, apply the generic camera contract and flag for human review.

  health_endpoint: true

- **Health**: Is the agent running? Is the FTP endpoint reachable? Are watch paths accessible?
- **Metrics**: Files ingested (count, rate), files quarantined, average ingest latency, checksum computation time, current queue depth.

## Non-Goals (v0.1)

- No active editing, transcoding, or color processing of media.
- No direct user-facing UI; UI is delegated to higher-level portals and dashboards.
- No long-term archival decisions; defer to archival agents and Storage Contracts.
- No AI tagging or classification; defer to downstream TAGGING agents triggered by ingest events.
- No multi-site replication; defer to SYNC agents.

## Constitutional Compliance

This agent operates under the authority of STORAGE.md and the main Constitution:

- **Article XXI**: Respects sovereign storage pools; never moves data between pools without contract authority.
- **Article XXII**: Consults Storage Contracts before every write operation.
- **Article XXIV**: Implements the first-class ingest process with all required steps.
- **Article XXVII**: Emits observable events for every ingest and error.
- **Article IV** (Separation of Powers): CAMERA_INGEST ingests but does not validate its own output for downstream quality; that responsibility belongs to other agents.
- **Article VIII** (Honesty): Reports errors immediately; never silently drops files.
- **Article IX** (Efficiency): Processes files as fast as the transport allows; does not introduce unnecessary latency.
- **Article XVII** (Compound Learning): Contributes ingest patterns and anomalies to the Live Memory Bus via SCRIBE.
